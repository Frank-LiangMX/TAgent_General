# TAgent 上游能力对齐规划（v0.10.34 基线）

> **状态**：Draft v0.1  
> **日期**：2026-06-16  
> **适用范围**：TAgent Desktop（`apps/electron`）  
> **目标**：在不破坏双模式架构的前提下，引入上游已验证的高价值能力

---

## 1. 背景与目标

当前 TAgent 以 `v0.10.34` 能力基线演进。上游在后续版本持续迭代了调度自动化、预览体验、模型上下文能力统一、稳定性修复等能力。

本规划的目标是：

- 将可复用且高价值的能力引入 TAgent；
- 避免直接搬运导致的模式隔离破坏（general/ta 混用）；
- 给开发 Agent 一份可执行的实施与验收手册，减少偏航和返工。

---

## 2. 范围与优先级

### 2.1 P0（优先落地）

1. **模型上下文能力统一**
   - 建立 `supports1MContext` / `inferContextWindow` 单一事实源（shared 层）。
   - 消除主进程、渲染层、设置页、渠道测试之间的窗口不一致。

2. **Automation 调度系统（最小可用）**
   - 支持 `interval` / `daily` / `weekly` 三种调度。
   - 后台调度、失败退避、启停管理、历史运行记录。
   - 数据持久化与迁移策略（版本化）。

3. **上下文安全阀 + 会话复用策略**
   - 调度任务执行前检查会话上下文占用阈值（建议 70%）。
   - 达阈值自动开新子会话，降低超长上下文风险。

### 2.2 P1（次优先）

1. **预览分屏模式**
   - 增加「标签页 / 分屏」偏好并持久化。
   - 保持现有预览路径兼容，不破坏当前 tab 体验。

2. **调度 monthly 模式**
   - 月度第 N 天调度，短月自动落最后一天。

3. **自动化表单保存状态指示**
   - `idle/dirty/saving/saved/error` 五态反馈。

### 2.3 P2（体验增强，可后置）

- 全局视觉 token 与原子组件重构（仅在功能稳定后进行）。
- 侧边栏信息架构微调（严格以 TAgent 双模式为前提，不照搬上游结构）。

---

## 3. 非目标（本轮不做）

- 不调整 TAgent 双顶层模式信息架构（general/ta 互斥原则不变）。
- 不引入破坏现有数据目录的重构（`~/.tagent/` 兼容优先）。
- 不进行大规模 UI 视觉重绘（避免与功能迭代耦合）。

---

## 4. 设计原则

1. **模式隔离优先**
   - general 与 ta 的会话、工具、记忆、状态不能交叉污染。

2. **共享能力下沉**
   - 通用逻辑放 `packages/shared` 或共用服务层，避免多处硬编码。

3. **可回滚**
   - 关键功能必须具备开关或降级路径。

4. **本地优先**
   - 持久化优先 JSON/JSONL/配置文件，避免引入新数据库方案。

5. **小步提交**
   - 按 GitHub Flow 分批实现，保持每个 PR 可独立验证。

---

## 5. 里程碑与交付

## M1：上下文能力统一（P0）

**交付物**

- 统一能力函数（shared）。
- 渲染层模型列表与主进程连通性测试口径一致。
- 单测覆盖关键映射逻辑。

**验收标准**

- 同一模型在设置页、会话进度显示、连通性测试结果一致。
- 无硬编码散落在多个模块。

## M2：Automation 最小可用（P0）

**交付物**

- 调度器服务（30s tick 或等效机制）。
- CRUD + 启停 + 运行历史。
- `interval/daily/weekly` 可执行。

**验收标准**

- 任务可创建、暂停、恢复、删除、手动运行。
- 应用重启后任务状态与下次运行时间正确恢复。

## M3：上下文安全阀（P0）

**交付物**

- 运行前占用率检查逻辑。
- 达阈值自动分新会话策略。

**验收标准**

- 高占用任务不会继续堆叠到同一会话。
- 日志与 UI 可见切换原因。

## M4：预览分屏与 monthly（P1）

**交付物**

- 预览模式偏好持久化。
- monthly 调度与短月边界处理。

**验收标准**

- 分屏/标签页切换稳定，无状态丢失。
- monthly 在 2/4/6/9/11 月行为符合预期。

---

## 6. 风险与应对

1. **风险：调度任务导致资源泄漏**
   - 应对：后台任务生命周期统一管理；超时与并发上限；退出时清理。

2. **风险：模式状态串线**
   - 应对：明确 atom/service 命名空间；新增测试覆盖模式切换场景。

3. **风险：UI 迭代影响现有交互**
   - 应对：先功能后样式；新增功能默认保守开关。

4. **风险：历史数据迁移损坏**
   - 应对：索引版本化；仅追加迁移；失败时只读降级，禁止清空。

## 7. 开发 Agent 实施手册（强制遵守）

本节是给后续开发 Agent 的操作说明，按此执行即可。

### 7.1 开发前检查

1. 阅读：
   - `AGENTS.md`
   - `.claude/CLAUDE.md`
   - `docs/PROGRESS.md`
2. 确认当前分支不是 `main`。
3. 明确本次任务属于哪个里程碑（M1~M4）。

### 7.2 分支与提交规范

- 分支：`feature/<scope>-<topic>` 或 `fix/<scope>-<topic>`
- 提交：Conventional Commits  
  例：`feat(desktop): add automation scheduler core loop`

### 7.3 实施步骤（每个子任务都按此流程）

1. **先写最小设计说明**
   - 影响文件
   - 数据结构变更
   - 回滚方式

2. **再实现**
   - 优先复用现有服务/atom
   - 禁止复制粘贴式重复实现

3. **补测试**
   - 共享逻辑、调度逻辑、迁移逻辑优先
   - 边界条件必须有用例（时区、短月、重启恢复）

4. **本地验证**
   - `bun run typecheck`
   - `bun test`
   - 必要时补充人工验证步骤

5. **更新文档**
   - 若行为变更，更新 `docs/PROGRESS.md`
   - 若架构变更，补 ADR 或更新设计文档

### 7.4 代码约束

- 禁止引入 `any`。
- 对象结构优先 `interface`。
- 状态管理统一用 Jotai。
- 注释与日志优先中文。
- 品牌统一使用 **TAgent**。

### 7.5 目录与职责建议

- 共享能力函数：`packages/shared/src/*`
- 主进程编排/调度：`apps/electron/src/main/lib/*`
- IPC 注册：`apps/electron/src/main/ipc.ts`
- preload 桥接：`apps/electron/src/preload/index.ts`
- 渲染状态：`apps/electron/src/renderer/atoms/*`
- 渲染 UI：`apps/electron/src/renderer/components/*`

新增 IPC 必须同步修改上述四层链路（shared 类型 → main handler → preload → renderer）。

### 7.6 PR 验收清单

- [ ] 功能点与里程碑目标一致
- [ ] 已补测试且通过
- [ ] 类型检查通过
- [ ] 未破坏 general/ta 隔离
- [ ] 文档已更新（至少 `docs/PROGRESS.md`）
- [ ] 提供回滚说明

## 8. 建议拆分的首批任务单（可直接派发）

1. **Task A（M1）**：提取并统一 `supports1MContext` / `inferContextWindow`
2. **Task B（M2）**：实现 Automation 数据结构、存储与基础调度循环
3. **Task C（M2）**：Automation 管理 UI（列表 + 新建 + 启停 + 手动运行）
4. **Task D（M3）**：上下文占用安全阀 + 新会话分流
5. **Task E（M4）**：预览分屏偏好与 monthly 调度

每个任务单独 PR，避免超大变更。

## 9. 时间建议（参考）

- M1：0.5~1 天
- M2：2~3 天
- M3：1 天
- M4：1~2 天

总计建议：**5~7 个工作日**（不含评审等待）。

## 10. 完成定义（Definition of Done）

满足以下条件视为本轮规划完成：

- P0 全部落地并通过测试；
- 至少完成 1 轮真实任务演练（自动任务创建 → 执行 → 结果检查）；
- 文档与代码状态一致；
- 未引入跨模式回归问题。
